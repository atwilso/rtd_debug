\doxysubsection{Detailed Description}
\subsubsection*{template$<$typename value\+\_\+type, typename rtree\+\_\+algorithm\+\_\+type = bgi\+::quadratic$<$16$>$$>$\newline
class tracktable\+::\+RTree$<$ value\+\_\+type, rtree\+\_\+algorithm\+\_\+type $>$}
This is a wrapper for the Boost rtree implementation in boost\+::geometry\+::index. 

Its purpose is to insulate you, the user, from having to care about all the complexity (and power) involved in boost\+::geometry\+::index\+::rtree. You supply a value type (which can be a point, a pair or a tuple) and we do the rest.

The disadvantage is that you\textquotesingle{}re restricted from using some of the more powerful query capabilities, including user-\/defined predicates and query combination.

\end{DoxyCode}


(your results vector now contains all the points inside the box)

You can populate this R-\/tree with any point type known to Tracktable / boost\+::geometry. This includes base\+\_\+point\+\_\+type and trajectory\+\_\+point\+\_\+type from all the domains as well as any other point that you have registered with boost\+::geometry.

You can also populate the R-\/tree with {\ttfamily std\+::pair\texorpdfstring{$<$}{<}\texorpdfstring{$>$}{>}}s and {\ttfamily boost\+::tuple\texorpdfstring{$<$}{<}\texorpdfstring{$>$}{>}}s where the first element of the pair / tuple is one of the point types listed above. Note that when you query the r-\/tree, {\itshape only} the geometry will be used in the query\+: the auxiliary information is preserved but never compared.

When querying the r-\/tree, you must use the same geometric point type that you used to populate it. Like the values stored in the tree, you can specify the search point(s) as a point type, a {\ttfamily std\+::pair\texorpdfstring{$<$}{<}\texorpdfstring{$>$}{>}} or a {\ttfamily boost\+::tuple\texorpdfstring{$<$}{<}\texorpdfstring{$>$}{>}} whose first element is the geometry. This is just for convenience\+: the only part that will be used for the search is the geometry.

You may only modify the contents of the R-\/tree with {\ttfamily \doxylink{classtracktable_1_1_r_tree_a1c0d28fa8b5ab6e1bed885c61608d5a4}{insert()}}, {\ttfamily \doxylink{classtracktable_1_1_r_tree_ad5378e07e120012a956938277ecf3583}{remove()}} and {\ttfamily \doxylink{classtracktable_1_1_r_tree_af37c8a3e169e5959bbe7c9dfcdfc2377}{clear()}}. There is no way to get a reference to an internal element and modify it directly as you can do with std\+::vector, std\+::map and friends. Doing so would break the search structure.

As with all the other \doxylink{classtracktable_1_1_r_tree}{RTree} functions, you can use a point type, a {\ttfamily std\+::pair\texorpdfstring{$<$}{<}point\+\_\+type, X\texorpdfstring{$>$}{>}} or a {\ttfamily boost\+::tuple\texorpdfstring{$<$}{<}point, type, \mbox{[}other stuff\mbox{]}\texorpdfstring{$>$}{>}} for your searches. \begin{DoxyWarning}{Warning}
This function is sensitive to numerical imprecision issues when points are (allegedly) right on the border of the search box. This is especially problematic in the terrestrial domain (longitude/latitude points) since we have to do trigonometry to compute point-\/in-\/polygon results.
\end{DoxyWarning}

Find points inside a search box (iterator range version) 

This function finds points inside a box specified as a \doxylink{classtracktable_1_1_box}{tracktable\+::\+Box} (also known as a tracktable\+::\+RTree$<$point\+\_\+type$>$\+::box\+\_\+type).

Example\+:


\begin{DoxyCode}{0}

\DoxyCodeLine{my\_point\ min\_corner,\ max\_corner;}
\DoxyCodeLine{}

\DoxyCodeLine{my\_tree.intersects(min\_corner,\ max\_corner);}
\DoxyCodeLine{}
\DoxyCodeLine{std::vector<my\_point>\ results(result\_range.first,\ result\_range.second);}

\end{DoxyCode}


Note that this function will return points that are exactly on the boundary of the search box as well as those in the interior. If you want only the points in the interior, use {\ttfamily find\+\_\+points\+\_\+strictly\+\_\+inside\+\_\+box}.

As with all the other \doxylink{classtracktable_1_1_r_tree}{RTree} functions, you can use a point type, a {\ttfamily std\+::pair\texorpdfstring{$<$}{<}point\+\_\+type, X\texorpdfstring{$>$}{>}} or a {\ttfamily boost\+::tuple\texorpdfstring{$<$}{<}point, type, \mbox{[}other stuff\mbox{]}\texorpdfstring{$>$}{>}} for your searches. In the case of a {\ttfamily std\+::pair} or {\ttfamily boost\+::tuple}, your geometry type must be the first element.

\begin{DoxyWarning}{Warning}
This function is sensitive to numerical imprecision issues when points are (allegedly) right on the border of the search box. This is especially problematic in the terrestrial domain (longitude/latitude points) since we have to do trigonometry to compute point-\/in-\/polygon results.
\end{DoxyWarning}
